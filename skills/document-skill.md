本技能指导生成软件模块的完整中文技术文档。产出三大核心交付物：**功能说明文档**、**接口清单**、**详细接口规格说明**（含模拟请求与模拟响应示例）。

用户提供模块或项目路径，技能分析源代码后生成结构化文档。

## 工作流程

### 第一阶段：代码分析

生成文档前，必须深入探索目标模块：

1. **项目结构**：识别所有子模块、目录及关键配置文件（pom.xml、build.gradle、package.json 等）
2. **启动入口**：找到主应用类、启动配置和引导文件
3. **控制层/路由层**：找到所有 REST 控制器、路由处理器或 API 端点定义
4. **服务层**：识别业务逻辑服务及其职责
5. **数据模型层**：阅读实体类、DTO、请求/响应模型
6. **配置信息**：阅读 application.yml/properties、环境配置和部署描述符
7. **接口契约**：找到接口定义、Feign 客户端、gRPC proto 文件或 OpenAPI 规范

**关键要求**：必须阅读实际源代码文件。禁止猜测或假设 API 签名。每个文档中的参数、返回类型和端点路径都必须从源代码中验证。

### 第二阶段：文档生成

生成一份包含三大部分的 Markdown 文档，输出到目标模块同级目录，命名为 `{module-name}-api-document.md`。

## 文档质量标准

### 参数文档规范
- 每个参数必须包含：参数名、类型、是否必填、参数位置（Query/Body/Path）、说明
- 枚举类型参数需列出所有可选值
- 嵌套对象需递归展开所有字段
- 标注默认值（如有）

### 模拟数据规范
- 使用贴近业务的真实示例值，禁止使用 "string"、"test"、"xxx" 等占位符
- 中文业务系统使用中文示例数据（如：名称用"一般公共预算"、编码用"101001"、机构名用"财政局"等）
- 模拟响应结构必须与声明的返回类型完全匹配
- 核心接口需同时提供成功和失败的响应示例
- 列表接口的响应中包含 2-3 条示例数据
- 分页接口的响应中包含分页元数据（total、page、rows）

### 命名规范
- 接口名称使用描述性中文命名："{动作}{资源}" 格式（如："新增要素"、"查询要素列表"、"根据ID查询要素"）
- 控制器和方法名保持源代码原文，不翻译代码标识符
- 按控制器分组组织相关接口

### 完整性要求
- 必须文档化控制器中找到的**所有**公开 API 端点，不得遗漏
- 文件上传/下载接口需包含 multipart 参数详情
- 若存在 Feign 客户端接口，需单独记录其额外契约
- 标注代码中可见的认证/授权要求（如 @RequiresPermissions、@PreAuthorize）

## 输出语言

- 文档正文全部使用**中文**输出
- API 路径、参数名、类名、方法名等代码标识符保持原文，不翻译
- 技术术语首次出现时可附英文原文（如：要素值集（Element Value Set））

## 参考实现

`element-module` 项目（`d:\CTJSOFT\Qoder\refactor\element-module`）作为参考实现，观察到的关键模式：

- **标准 CRUD 模式**：POST /add 新增、PUT /update 修改、DELETE /delete 删除、GET /getList 查询列表、GET /getById 按ID查询
- **统一返回类型**：`ReturnData<T>` 包含 status_code、reason、data 三个字段
- **分页查询**：PUT 方法 + page/rows 参数 + @RequestBody 过滤条件，返回 `ReturnData<ReturnPage>`
- **树形查询**：GET 端点返回层级列表结构
- **文件操作**：POST + MultipartFile 导入，POST + HttpServletResponse 导出
- **Feign 契约**：接口继承控制器接口并标注 `@FeignClient` 注解
- **多维查询**：getById、getByCode、getByXxx 提供同一资源的多种访问路径

分析其他模块时，识别类似模式并据此调整文档结构。
