# 修复安全约束与核心原则

## 核心原则

| 编号 | 原则 | 说明 |
|------|------|------|
| C-01 | **只做命名和规范调整，不改业务逻辑** | 仅修改类名、属性名、HTTP 方法兼容性，不修改方法实现内容 |
| C-02 | **约束优先** | 标记为"约束限制"的项不修改（URL、JSON 序列化兼容） |
| C-03 | **用户确认** | 所有修改计划须获得用户确认后执行 |
| C-04 | **保持可编译** | 修改后确保所有引用路径正确 |
| C-05 | **幂等性** | 重复执行不产生额外变更，已合规项自动跳过 |
| C-06 | **确定性** | 相同输入必须产生相同输出，不依赖随机性或外部状态 |

## 安全约束（红线）

执行修复前必须遵守以下约束，**任何一条违反都必须立即停止并报告**：

| 编号 | 约束 | 说明 |
|------|------|------|
| S-01 | **不修改 HTTP 接口 URL** | 除非用户明确要求，不得修改任何接口路径 |
| S-02 | **不修改 HTTP 方法** | 除兼容性增强（DELETE/PUT 添加 POST 兼容）外，不得变更 |
| S-03 | **不修改业务逻辑代码** | 任何方法体内容不得改变 |
| S-04 | **不修改 DTO/VO 属性名** | 除非配合 `@JsonProperty` 保持兼容 |
| S-05 | **必须用户确认** | 重构前必须将修复计划展示给用户并获得确认 |
| S-06 | **无文件冲突** | 目标目录不得存在同名文件，修改前必须检查 |
| S-07 | **编码保留** | 修改或重命名文件时必须保持原文件的字符编码格式（如 UTF-8、UTF-8 with BOM、GBK 等）。原文件为 UTF-8 with BOM（首3字节为 EF BB BF）时，修改后或重命名后的文件必须保留 BOM。类名重命名涉及文件重建时，优先使用 Bash 的 `mv` 命令重命名文件（保留编码），再用 Edit 修改文件内容；避免使用 Read → Write 方式重建文件导致编码丢失 |
| S-08 | **不引入新冲突** | 修复一个问题不得引入新的 Bean 冲突或命名冲突 |
| S-09 | **不修改外部接口契约** | Feign 接口、RPC 接口、对外暴露的 API 路径不得修改 |

## 允许修改的范围

- Java 类名（同步修改文件名和所有引用）
- `@DeleteMapping` / `@PutMapping` 兼容增强（添加 POST 兼容）
- Spring Bean 名称（显式指定以解决冲突）
- 类文件名（与类名保持一致）
- import 语句（因上述修改产生的联动更新）
- 注入点变量名（仅当变量名与旧类名直接对应时）

## 不允许修改的范围

- 任何 Java 方法体
- 任何接口 URL 路径
- DTO/VO 属性名（影响 JSON 序列化兼容性）
- 接口参数名（影响前端传参兼容性）
- 任何前端代码
- 已发布的接口 URL 和 HTTP 方法
- `@GetMapping`、`@PostMapping` 的 HTTP 方法
- `@PatchMapping`（仅报告，不修复）
- 枚举类名（跳过后缀检查）
- 工具类/配置类/常量类的层级后缀

## 幂等性保障规则

### 检查阶段幂等性

| 保障项 | 规则 |
|--------|------|
| 排序确定性 | 检查结果按编号 → 文件路径 → 行号排序 |
| 去重 | 同一问题（文件+行号+类型）不重复报告 |
| 状态无关 | 检查结果仅基于代码静态分析，不依赖时间/环境变量 |

### 修复阶段幂等性

| 保障项 | 规则 |
|--------|------|
| 前置检查 | 修复前先检查目标状态是否已达成 |
| 合规跳过 | 已合规项输出"已合规，跳过" |
| 无效操作抑制 | 不对已正确的代码执行任何修改操作 |
| 重复执行安全 | 技能多次执行后代码状态与首次执行后一致 |

### 幂等性检查清单

对每个修复项，在执行修复前必须先执行以下检查：

```
修复规范一（注解兼容）：
  ✓ @DeleteMapping 不存在于文件中？ → 跳过
  ✓ @PutMapping 不存在于文件中？ → 跳过
  ✓ 已是 @RequestMapping(method={DELETE/PUT, POST}) 形式？ → 跳过

修复规范二（类命名修正）：
  ✓ 类名后缀已合规？ → 跳过
  ✓ 类名已是大驼峰？ → 跳过
  ✓ 文件名已与类名一致？ → 跳过

修复规范三（Bean 冲突）：
  ✓ 全局无重复 Bean 名？ → 跳过
  ✓ "2" 后缀已不存在？ → 跳过
```

## 重复执行安全

### 场景：技能在同一工程上执行多次

| 执行次数 | 预期行为 |
|---------|---------|
| 第 1 次 | 检查 → 发现 N 个问题 → 修复 → 验证通过 |
| 第 2 次 | 检查 → 发现 0 个可修复问题 → 报告"所有项已合规" |
| 第 N 次 | 与第 2 次行为完全一致 |

### 场景：部分修复后再次执行

| 情况 | 预期行为 |
|------|---------|
| 上次修复了 3/10 项后中断 | 本次检查 → 发现剩余 7 个问题 → 仅对这 7 项生成修复计划 |
| 上次修复后有新代码提交 | 本次检查 → 仅对新增不合规代码生成修复计划 |

### 场景：修复后人工又改回去

| 情况 | 预期行为 |
|------|---------|
| 开发者改回了旧类名 | 本次检查 → 重新发现该问题 → 生成修复计划 |

## 操作前检查清单

每次执行修复前，确认以下前提条件：

```
□ S4（Controller 层治理）已完成
□ 工程可编译通过（或至少不会因本次修改引入新编译错误）
□ 用户已确认修复计划
□ 目标文件无其他人正在编辑（如可判断）
□ Git 工作区干净或已提交（建议在修复前提交当前代码）
```
