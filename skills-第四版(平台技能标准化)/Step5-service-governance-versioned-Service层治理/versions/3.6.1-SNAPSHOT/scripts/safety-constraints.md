# 修复安全约束与核心原则

## 核心原则

| 编号 | 原则 | 说明 |
|------|------|------|
| C-01 | **只做结构调整，不改业务逻辑** | 仅修改 package 声明和 import 语句，不修改方法实现内容 |
| C-02 | **安全迁移** | 先读取原文件 → 在新位置创建文件 → 更新引用 → 删除原文件 |
| C-03 | **约束优先** | 不修改跨模块共享接口的公共 API |
| C-04 | **用户确认** | 所有修改计划须获得用户确认后执行 |
| C-05 | **保持可编译** | 迁移后确保所有 import 路径和 package 声明正确 |

## 安全约束（红线）

执行修复前必须遵守以下约束，**任何一条违反都必须立即停止并报告**：

| 编号 | 约束 | 说明 |
|------|------|------|
| S-01 | **不修改业务逻辑代码** | 任何方法体内容不得改变 |
| S-02 | **不修改非 service 包** | 不影响 controller/dao/model 等其他包的结构 |
| S-03 | **不移动非 Service 文件** | 常量、枚举、工具、异常、Feign 等保留原位 |
| S-04 | **不修改接口定义** | Service 接口的方法签名不得改变 |
| S-05 | **必须用户确认** | 重构前必须将修复计划展示给用户并获得确认 |
| S-06 | **无文件冲突** | 目标目录不得存在同名文件，迁移前必须检查 |
| S-07 | **跨模块接口评估** | 跨模块共享的 Service 接口需评估影响范围后再执行 |

## 允许修改的范围

- Service 接口 Java 文件的 `package` 声明行
- Service 实现 Java 文件的 `package` 声明行
- 引用 Service 接口/实现的其他文件的 `import` 语句行
- 目录结构（创建 facade/impl 目录、移动文件、删除空目录）
- Spring 配置中的 `@ComponentScan` 路径（如因包路径变更需要）

## 不允许修改的范围

- 任何 Java 方法体
- 任何接口的方法签名
- 任何类/接口/枚举的定义结构
- 任何 Controller / DAO / Model 层的代码和结构
- 任何 XML 配置文件内容
- 任何前端代码
- 任何 pom.xml
- 非 Service 文件的位置（常量、枚举、工具、异常、Feign）
