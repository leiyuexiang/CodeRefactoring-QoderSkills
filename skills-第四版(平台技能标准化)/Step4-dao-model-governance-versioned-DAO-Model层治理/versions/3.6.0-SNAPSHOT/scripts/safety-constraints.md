# 修复安全约束与核心原则

## 核心原则

| 编号 | 原则 | 说明 |
|------|------|------|
| C-01 | **只动目录和包路径，不改业务逻辑** | 仅修改 package 声明和 import 语句 |
| C-02 | **安全迁移** | 先读取原文件 → 在新位置创建文件 → 更新引用 → 删除原文件 |
| C-03 | **逐步执行** | 按优先级逐项修复，每完成一项向用户确认 |
| C-04 | **保持可编译** | 迁移后确保所有 import 路径和 package 声明正确 |

## 安全约束（红线）

执行修复前必须遵守以下约束，**任何一条违反都必须立即停止并报告**：

| 编号 | 约束 | 说明 |
|------|------|------|
| S-01 | **不修改类名** | 任何类名、方法签名、接口定义不得改变 |
| S-02 | **不修改 HTTP 接口** | URL、HTTP 方法不得改变 |
| S-03 | **不修改 Entity 包名** | Entity 类的包名不得改变（除非用户明确要求迁移到 dao/entity/） |
| S-04 | **不修改业务逻辑** | 任何业务逻辑代码不得改变 |
| S-05 | **必须用户确认** | 重构前必须将修复计划展示给用户并获得确认 |
| S-06 | **文件冲突预检与处理** | 迁移前必须检查目标位置是否存在同名文件（详见下方冲突处理规则） |
| S-07 | **禁止创建 Service 接口或实现类** | Step4 的职责是 DAO/Model 层的目录结构治理。创建 Service 接口（如 IXxxDelegateService）、Service 实现类（如 XxxDelegateServiceImpl）属于 Step3（架构依赖守卫）或 Step5（Service层治理）的职责范围。Step4 绝不可新建任何 Service 类文件 |
| S-08 | **禁止修改 Controller 层代码** | Step4 不得修改 Controller 层的任何文件内容（包括 @Autowired/@Resource 注入、方法调用链等），仅允许更新 Controller 层文件中因 DAO/Model 文件迁移导致的 import 语句变更。Controller 层依赖关系调整属于 Step3 的职责范围 |
| S-09 | **禁止修改 pom.xml 依赖** | Step4 绝不可增删改 pom.xml 中的 `<dependency>` 条目。即使文件迁移导致编译问题，也只能在修复报告中标注，不得自行调整 POM 依赖 |
| S-10 | **Step4 职责范围明确定义** | Step4 仅允许执行以下操作：(1) 重命名 imp→impl 目录（含合成词变体如 serviceImp→impl） (2) 移动 DAO 实现类到 dao/impl/ (3) 将独立 mapper 包迁入 dao/mapper/ (4) 将 Entity 类归入 dao/entity/ (5) 将 model/ 散落文件按类名后缀归档到 dto/vo/query/po 子目录 (6) 创建缺失的标准子目录 (7) 更新上述迁移涉及的 package 声明和 import 语句。**任何超出上述范围的操作均视为越权，必须立即停止** |
| S-11 | **独立业务域模块豁免** | 满足独立业务域模块判定标准的目录（详见全局前置规则），其内部文件不做任何迁移或重命名。仅允许更新其中因其他位置文件迁移而受影响的 import 语句 |

## S-06 文件冲突预检与处理规则

### 迁移前冲突预检

在将任何文件迁移到新位置之前，**必须**使用 Glob 检查目标位置 `{target-dir}/{FileName}.java` 是否已存在同名文件。

### 冲突处理策略（确定性）

| 场景 | 条件 | 处理 |
|------|------|------|
| 无冲突 | 目标位置不存在同名文件 | 正常执行迁移（Read → Write → Update imports → Delete） |
| 完全相同 | 目标位置存在同名文件，且与源文件内容一致 | 跳过创建新文件，仅删除源文件并更新所有 import 指向目标位置 |
| 内容不同 | 目标位置存在同名文件，内容不同 | **停止本文件迁移**，在修复报告中标注为 **CONFLICT**，不做任何操作，等待人工介入 |

### 内容比较方法

读取源文件和目标文件的全部内容，**忽略以下行的差异**后逐行比较：
- `package` 声明行（因包路径不同导致的预期差异）
- AI 标记注释行（`// @AI-Begin`、`// @AI-End` 等）

## import 搜索范围定义

"全代码库"的精确定义为：**当前被修复模块的顶级父 POM 所在目录及其所有子目录**。

**确定搜索根目录的算法**：
1. 从被修复文件所在的 Maven 模块 `pom.xml` 开始
2. 向上追溯 `<parent>` 关系，直到找到最顶层的 `pom.xml`（没有 `<parent>` 或 `<parent>` 指向远程仓库）
3. 该顶层 `pom.xml` 所在目录即为搜索根目录
4. 在搜索根目录下递归 Grep 搜索旧包路径

**排除目录**：搜索时排除 `target/`、`.git/`、`node_modules/`、`.qoder/` 目录

**特殊说明**：如果项目根目录存在一个聚合 POM 同时包含多个模块目录作为 `<modules>`，则搜索范围扩大到该聚合 POM 所在目录。**此判定在 Phase 1 扫描阶段一次性确定，修复过程中保持不变。**

## 允许修改的范围

- Java 文件的 `package` 声明行
- Java 文件的 `import` 语句行
- 目录结构（创建新目录、移动文件、删除空目录）
- MyBatis mapper XML 中的 `namespace`（因包路径变更）
- `@MapperScan` 注解的 `basePackages`（因包路径变更）
- Spring 配置中的 `@ComponentScan` 路径（如因包路径变更需要）

## 不允许修改的范围

- 任何 Java 方法体
- 任何类/接口/枚举的定义结构
- 任何 XML 配置文件内容（MyBatis mapper XML 的 namespace 除外）
- 任何前端代码
- 任何 pom.xml 的 `<dependency>` 部分（绝对禁止）
- 任何 Controller 层文件的业务逻辑（仅允许更新因迁移导致的 import 语句）
- 任何 Service 层的业务逻辑文件（仅允许更新被迁移文件的 import 引用）
- 不得新建任何 Service 接口或 Service 实现类文件
- 独立业务域模块内的文件（仅允许更新 import 语句）
