# Qoder 前端规范技能执行标准流程

## 概述

本文档定义了前端工程规范检查与治理的 **Qoder 技能执行标准流程**。整个流程包含 5 个技能，按固定顺序依次执行，从项目结构治理逐步细化到命名规范检查，形成完整的前端工程治理链路。

**核心原则**：所有技能均不改变业务逻辑，只调整目录结构、组件规范、页面布局、命名规范和代码质量。

---

## 执行顺序总览

| 序号 | 技能名称 | 优先级 | 作用层级 | 核心职责 |
|------|----------|--------|---------|----------|
| Step1 | 项目结构治理 | -- | 工程级 | 前端项目顶层目录结构检查与治理 |
| Step2 | 模块划分治理 | -- | 工程级 | 业务模块划分、内部结构、模块间依赖检查与治理 |
| Step3 | UI 组件规范 | 重要 | 代码级 | 颜色、字体、按钮、录入、表格规范检查与代码生成 |
| Step4 | 布局与交互规范 | 重要 | 代码级 | 页面布局、层级、弹窗、抽屉、交互规范检查与代码生成 |
| Step5 | 命名规范检查 | 优化 | 代码级 | 组件、文件、变量、CSS 类名等命名规范检查与修复 |

---

## 流程依赖关系

```
Step 1: 项目结构治理（标准顶层目录搭建）
    │
    ▼
Step 2: 模块划分治理（业务模块结构规范化，依赖 Step 1 产出的标准目录结构）
    │
    ▼
Step 3: UI 组件规范（组件级规范检查，依赖 Step 2 完成后的模块结构）
    │
    ▼
Step 4: 布局与交互规范（页面级规范检查，依赖 Step 3 组件规范已就绪）
    │
    ▼
Step 5: 命名规范检查（全局命名检查，依赖 Step 1~4 结构全部就绪）
```

## 各步骤详细说明

### Step 1: 项目结构治理

**技能全名**：`/Step1-project-structure-versioned-项目结构治理`

**触发关键词**：项目结构检查、目录规范、分层架构审查、项目初始化、脚手架生成、项目结构治理

**核心职责**：
- 检查 src/ 下 7 个标准顶层目录是否存在（assets/components/modules/framework/services/utils/types）
- 检查 components/ 三层分类（common/layout/business）
- 检查 framework/ 内部结构（router/store/plugins 等）
- 检查跨层引用方向是否合法
- 提供项目脚手架模板快速初始化标准结构

**检查项清单**：

| 编号 | 检查项 | 严重级别 |
|------|--------|--------|
| S9-01 | 顶层目录完整性 | ERROR |
| S9-02 | assets 目录结构 | WARNING |
| S9-03 | components 分层 | ERROR |
| S9-04 | 复合组件目录形式 | WARNING |
| S9-05 | framework 目录结构 | WARNING |
| S9-06 | services 目录结构 | WARNING |
| S9-07 | composables 命名 | ERROR |
| S9-08 | 禁止跨层引用 | ERROR |

---

### Step 2: 模块划分治理

**技能全名**：`/Step2-module-division-versioned-模块划分治理`

**触发关键词**：模块划分检查、模块结构治理、业务模块拆分、模块依赖检查、创建业务模块

**核心职责**：
- 检查各模块内部结构完整性（views/api/types/index.ts）
- 检查模块大小（views/ 文件数 ≤ 10）
- 检查模块命名规范（kebab-case）
- 检查模块间依赖方向
- 提供模块脚手架模板快速创建标准模块

**前置条件**：
- **必须先执行 Step 1**，确保项目已具备标准顶层目录结构

**检查项清单**：

| 编号 | 检查项 | 严重级别 |
|------|--------|--------|
| S10-01 | 模块内部结构完整性 | ERROR |
| S10-02 | 模块大小控制 | WARNING |
| S10-03 | 模块命名规范 | ERROR |
| S10-04 | 导出入口完整性 | ERROR |
| S10-05 | API 与 Types 对应 | WARNING |
| S10-06 | 子模块结构规范 | WARNING |
| S10-07 | 模块间依赖方向 | ERROR |
| S10-08 | 路由对应检查 | WARNING |
| S10-09 | Store 对应检查 | WARNING |
| S10-10 | 视图命名规范 | WARNING |

---

### Step 3: UI 组件规范

**技能全名**：`/Step3-ui-component-standard-versioned-UI组件规范`

**触发关键词**：UI组件规范、颜色检查、字体规范、按钮规范、表格规范、录入规范、组件样例、生成组件

**优先级**：重要 -- UI 组件实现必须符合一体化界面规范

**核心职责**：
- 颜色规范检查（品牌色、中性色、功能色）-- 5 条规则
- 字体排版检查（字号、字重、对齐）-- 5 条规则
- 按钮规范检查（主按钮数量、更多面板）-- 8 条规则
- 录入控件检查（必填标记、标签布局）-- 8 条规则
- 表格规范检查（行号列、冻结列、对齐）-- 11 条规则
- 提供 7 类组件标准样例和 6 种组件模板

---

### Step 4: 布局与交互规范

**技能全名**：`/Step4-layout-interaction-versioned-布局与交互规范`

**触发关键词**：页面布局规范、交互规范、页面层级、弹窗规范、抽屉规范、滚动条规范、生成页面、页面模板

**优先级**：重要 -- 页面布局和交互必须符合一体化界面规范

**核心职责**：
- 页面布局检查（层级规范、面包屑、间距）-- 9 条规则
- 交互行为检查（滚动条、悬浮、提示消息）-- 8 条规则
- 提供 7 类页面布局样例和 4 种页面级模板

---

### Step 5: 命名规范检查

**技能全名**：`/Step5-naming-convention-versioned-命名规范检查`

**触发关键词**：命名规范、组件命名、文件命名、变量命名、CSS类名、命名检查、命名审查

**优先级**：优化 -- 命名规范是最后的代码质量检查

**核心职责**：
- 组件命名检查（PascalCase）
- 文件命名检查（kebab-case / PascalCase）
- 变量命名检查（camelCase / UPPER_SNAKE_CASE）
- CSS 类名检查（BEM 或 CSS Modules）
- 事件函数命名检查（handle/on 前缀）
- Props 类型命名检查（组件名 + Props 后缀）

**前置条件**：
- **建议先执行 Step 1 至 Step 4**，确保结构和规范已就绪

**检查项清单**：

| S8-01 | 组件命名规范 | ERROR |
| S8-02 | 文件命名规范 | ERROR |
| S8-03 | 变量命名规范 | WARNING |
| S8-04 | CSS 类名规范 | WARNING |

## 执行注意事项

### 通用安全约束

1. **不改变业务逻辑**：所有技能只做结构调整和规范检查，不修改业务代码行为
2. **用户确认机制**：每个技能执行修改前必须生成修复计划，等待用户确认后才执行
3. **备份提醒**：执行结构调整前确认用户已备份工程
4. **引用更新**：移动文件后必须同步更新所有 import 路径

### 执行顺序不可打乱的原因

| 依赖关系 | 说明 |
|---------|------|
| Step 2 依赖 Step 1 | 模块划分基于 Step 1 产出的标准顶层目录结构 |
| Step 3 依赖 Step 2 | 组件规范检查需在模块结构规范后执行 |
| Step 4 依赖 Step 3 | 页面布局依赖组件规范已确立 |
| Step 5 依赖 Step 1~4 | 命名规范是最后的全局检查，依赖所有结构调整完成 |

### 单步执行 vs 全流程执行

- **全流程执行**：按 Step 1 ~ Step 5 顺序逐步执行，适用于全新项目初始化或全面规范审查
- **单步执行**：可单独执行某个 Step，但需确保其前置 Step 已完成
- **检查+治理**：每个 Step 均支持"仅检查"、"仅治理/生成"、"检查并治理"三种模式
