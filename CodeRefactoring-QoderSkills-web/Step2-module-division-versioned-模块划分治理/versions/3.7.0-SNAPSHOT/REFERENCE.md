# S10 模块划分治理规则（3.6.0-SNAPSHOT）

## 概述

本文档定义了前端业务模块划分检查与治理的完整规则集。基于 framework-web2-server 项目分包规范，对前端项目的业务模块内部结构、模块大小、模块命名、模块间依赖、路由与 Store 对应关系等进行标准化约束。

**本技能提供两大功能：**
1. **模块划分检查**：扫描 modules/ 目录，发现不合规的模块结构和划分问题
2. **模块划分治理**：根据检查结果自动或辅助调整模块结构；或通过脚手架模板创建标准模块

## 使用场景

| 用户意图 | 触发关键词 | 执行功能 |
|---------|-----------|----------|
| 验证模块划分是否合规 | "模块划分检查"、"模块结构检查"、"模块依赖检查" | 功能一：模块检查 |
| 创建新业务模块 | "创建业务模块"、"新建模块"、"模块脚手架" | 功能二：脚手架生成 |
| 治理模块划分问题 | "模块结构治理"、"业务模块拆分"、"模块整改" | 功能一 + 功能二：检查 + 治理 |

## 检查项总览

| 编号 | 检查项 | 严重级别 | 说明 |
|------|--------|---------|------|
| S10-01 | 模块内部结构完整性 | ERROR | 每个模块必须含 views/、api/、types/、index.ts |
| S10-02 | 模块大小控制 | WARNING | 单模块 views/ 文件数不超过 10 个 |
| S10-03 | 模块命名规范 | ERROR | 模块目录名使用 kebab-case |
| S10-04 | 导出入口完整性 | ERROR | 每个模块必须有 index.ts 导出入口 |
| S10-05 | API 与 Types 对应 | WARNING | api/ 与 types/ 中的文件应一一对应 |
| S10-06 | 子模块结构规范 | WARNING | 子模块应遵循与模块相同的结构 |
| S10-07 | 模块间依赖方向 | ERROR | 禁止直接引用其他模块内部文件 |
| S10-08 | 路由对应检查 | WARNING | 每个模块应有对应的路由配置 |
| S10-09 | Store 对应检查 | WARNING | 使用状态管理的模块应有对应 store |
| S10-10 | 视图命名规范 | WARNING | views/ 下的页面组件使用 PascalCase |

## 模块划分标准

### 划分原则
- 按**业务域**划分模块，不按页面类型
- 模块划分决策矩阵：业务独立性、数据边界、功能聚合度、团队归属、复用性（满足 3 项及以上独立为模块）

### 标准模块内部结构

```
modules/{module-name}/
├── views/               # 页面视图
│   ├── ListView.vue      # 列表页
│   ├── DetailView.vue    # 详情页
│   └── FormView.vue      # 表单页
├── components/           # 模块私有组件
├── composables/          # 模块组合式函数
├── api/                  # 模块 API 接口定义
│   └── index.ts
├── types/                # 模块类型定义
│   └── index.ts
└── index.ts              # 模块导出入口
```

### 模块大小限制
- 单模块 `views/` 文件数 ≤ 10 个
- 超过时按子业务域拆分为子模块

### 依赖方向约束
```
modules/A → components/     ✅   modules/A → modules/B/内部   ❌
modules/A → services/       ✅   modules/A → modules/B/api/   ❌（应通过共享层中转）
modules/A → utils/          ✅
```

## 功能一：模块划分检查流程

### Phase 1: 扫描模块目录
1. 扫描 `src/modules/` 下所有子目录
2. 识别各模块的内部结构

### Phase 2: 逐项检查
按 S10-01 至 S10-10 的规则逐项检查，详见 `scripts/check-rules.md`

### Phase 3: 生成检查报告
输出合规项、违规项、修复建议

### Phase 4: 建议修复方案
针对每个违规项提供具体修复建议

---

## 功能二：模块脚手架生成流程

### Phase 1: 确认模块信息
1. 确认模块名称（kebab-case）
2. 确认所属业务域
3. 确认是否需要子模块

### Phase 2: 生成标准模块结构
使用 `templates/module-structure-template.md` 模板生成标准模块目录和基础文件

### Phase 3: 注册路由和导出
1. 在 `framework/router/modules/` 添加路由配置
2. 更新模块导出入口

### Phase 4: 验证结构
使用 S10-01 至 S10-10 规则验证生成的模块结构是否合规

---

## 安全约束

1. **不改变业务逻辑**：仅调整模块结构和文件位置
2. **用户确认机制**：执行模块拆分前必须生成计划，等待用户确认
3. **引用更新**：移动文件后必须同步更新所有 import 路径
4. **路由同步**：模块结构变更后必须同步更新路由配置
